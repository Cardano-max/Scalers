## Scalers P1.5 Design-Hardening Memo

**Overall verdict: the blueprint's direction is CORRECT on all five pillars, but each is naive in load-bearing ways that would produce decorative or dishonest artifacts if built literally. None require a rethink; all require specific corrections. The unifying failure mode is "object bolted on top of unchanged inline behavior" — a plan/board/entity that exists but nothing consumes, or a model/consent/progress value that is recorded but not measured. Every pillar's fix is: make the new artifact load-bearing and honest-by-construction, not additive.**

### A — Planner/Executor split (CampaignBlueprint + plan_campaign): SOUND, correct-these
The real defect is real: `_plan_node` (compose.py:153-160) is decorative and the LIVE tattoo path's strategist emits only a single `target_angle` string (agui.py:1198-1222). But the design targets the wrong spine. **The live tenant ladies8391 runs `_execute_provided_leads_sync` (agui.py:1101), a separate executor that never touches compose** — so plan_campaign must sit in the dispatcher `_execute_campaign_sync` (~agui.py:1063) BEFORE the provided-vs-compose branch, or the plan stays decorative for the actual tenant. Two hard gates ride on this: (1) the provided-leads path has NO fan-out cap on uploaded CSV `customer_ids` (agui.py:1154-1158) — only compose has `_OUTPUT_HARD_CAP=12` (compose.py:252) — a 5000-row CSV = 5000×(analyst+draft+critic) real calls, a cost-runaway AND exactly-once violation; (2) `offer_logic` must be built from `get_offers()+substantiate()` at plan time while the per-lead `substantiate()` fail-closed gate (agui.py:1299-1301) REMAINS. Run plan_campaign ONCE; defer replanning to Pillar B (do NOT copy Plan-and-Act's replan-every-step; match LangChain's replan-only-on-failed-effect baseline).

### B — Durable progress board + progress-aware replanning: SOUND, correct-these
Board-as-projection over the three existing store seams (agui.py:473/488/499), firing only on a measured contradiction, is exactly right (PABU, Task-Decoupled Planning, VLAs-as-Tools; reflect-every-step is documented to degrade). Three corrections are non-negotiable: (1) **"objections_resolved" is a fabrication landmine** — nothing sends in P1.5, no reply loop exists, so nothing is ever "resolved"; rename to `objections_addressed` = objection-matched STAGED drafts. (2) The **replan trigger MUST guard on sample_n, margin, assumed==measured short-circuit, and REPLAN_CAP**, or one noisy analyst read thrashes the plan — "limited commitment" is unimplemented without these. (3) **Contradiction must be an object** (assumed/measured/measured_count/sample_n/margin/evidence_run_ids) and `maybe_replan` must return None unless it emits a concrete PlanDelta — a decorative "replanned" note must be impossible by construction. Refactor `build_progress_context` (agui.py:514) into `compute_progress_board() + a renderer` so there is ONE counting path. Persist only the replan EVENT as a `record_agent_run` row with a DETERMINISTIC id (ON CONFLICT DO NOTHING, store.py:115) — no mutable board table. Requires a new `assumed_objection` field on CampaignPlan, populated by Pillar A.

### C — Compute routing + prompt caching: SOUND, but fix a pre-existing truthfulness bug FIRST
Routing needs no new plumbing (Cell already takes `model=`, base.py:123/223/234; palette pinned in test_config.py:29-31). **There is a live truthfulness bug that Pillar C will amplify: agent_run.model is HARDCODED, not read off the cell, for draft/analyst/strategist-fail in the provided-leads path (agui.py:1210/1272/1308-1312).** The draft literal `claude-sonnet-4-6` already contradicts the Opus-pinned copywriter (draft.py:41) — the trace is untruthful TODAY. The correct `_cell_model()` pattern already exists (agui.py:1187-1189; "read off the cell, never hardcoded" at contentrun.py:204, orchestrator.py:192). Caching has zero seam (bare `model_settings={'temperature':...}`, base.py:162-176) and the naive version is a NO-OP/net-negative: stable instruction blocks (~100-150 tokens) are far below Anthropic's 1,024-token (Sonnet/Opus) / 4,096-token (Haiku 4.5) minimums, and cache writes cost 1.25x. Caching only pays if the LARGE stable docs (brand playbook + offers doc + taxonomy + tool schema) are hoisted into the cached prefix, and must be proven by real `cache_read_tokens>0`, not configuration.

### D — Canonical object model: SOUND — the risk is OVER-BUILD, not a missing foundation
Most of this already exists and works: canonical dataclasses (Lead, ConversationThread, Artist, Offer), adapter Protocols with honest CSV/DB impls + NotConfiguredError stubs, and run-state ALREADY grounded in durable stores not transcript replay (build_progress_context/run_narration read runs/agent_runs/actions). **Do NOT author a parallel entities.py of pydantic models — it forks the honest dataclass contracts and risks an import cycle (adapters↔entities).** Make entities.py re-export/own the EXISTING types. Precise the stub story: Asset(content) is REAL (team/store.py:53) — do not stub it; Artist has real CSV/seed loaders; only Shop (no model at all), the Mini-App CRM/artist API, and Asset(artwork/VLM) are genuinely unconnected. **The real modeling gap is Consent** — currently an embedded boolean (email_opt_in/sms_opt_in), which cannot audit the TCPA/A2P SMS HELD gate (spec.py:106); promote it to a first-class provenance-bearing field (channel/basis/source/granted_at) and route the consent_required gate through it.

### E — Psych analyst as first-party progressive-disclosure skillpack: SOUND, but the naive placement TRIPS CI
Progressive disclosure (SKILL.md L1/L2 + references L3, deterministic logic stays code) is right, and keeping it IN-VETTING with the live path untouched is right. Two CI/fabrication traps: (1) `check_skill_registry.py` discovers bundles ONLY under `skills/*/` and `engine/skills/*/` (lines 48/145); an IN-VETTING (=HELD) row on a discovered bundle is a HARD FAIL (lines 204-209). **Staging at `engine/studio/skillpacks/customer_psychology/` is correct precisely because it is CI-blind (outside both roots) — the design must state this explicitly so nobody "helpfully" moves it into engine/skills/ and reddens the build.** (2) "a real 40-hex upstream pin" is WRONG for a first-party pack — there is no upstream, and inventing a 40-hex SHA is fabrication. Use `pinned: ORIGINAL` (copywriter/reply precedent; check supports it at lines 56-58/216-229); optionally anchor a real self-SHA of our frozen psych_profile.py in manifest.yaml labeled as content-integrity, not upstream. Also: frontmatter `name` must be `customer-psychology` (hyphens, no underscore, no reserved words). Registration into engine/skills/ + row flip to REGISTERED must be ONE atomic commit (lines 248-262). File a docs bead: the CLAUDE.md HARD RULE text ("real 40-hex upstream pin") must be reconciled to permit ORIGINAL for first-party packs, or registration is blocked/forces fabrication.

### How the pieces fit
Pillar A produces the blueprint (and `assumed_objection`) that Pillar B's contradiction engine reads; A's total_draft_cap and B's replan events share the same exactly-once idempotency discipline (deterministic ids + ON CONFLICT DO NOTHING). Pillar D's Consent entity is what makes A's/B's "nothing sends, SMS stays HELD" gate auditable. Pillar C's model-truth fix is a prerequisite for trusting every trace A/B/D produce. Pillar E rides alongside without touching any live path. All five share the same non-negotiable floor: HELD/approve-first on all sends, exactly-once, no fabrication (psychology/offers/numbers/sources/progress), our-own vetted skills only, frozen submodule eng5/src@699ef51 untouched, CI green.

## Open Risks
- CLAUDE.md HARD RULE wording ('a real 40-hex upstream pin that matches SKILL.md pinned:') cannot be satisfied by a first-party pack without fabricating an upstream SHA, even though the CI check and copywriter/reply rows already accept ORIGINAL. Until sec reconciles the runtime rule text (docs bead), registration is blocked or someone fabricates a pin. This is a policy/wording gap, not a code gap.
- The IN-VETTING->REGISTERED transition for the skillpack must be ONE atomic commit (relocate to engine/skills/ + flip row to REGISTERED — IN USE with separate sec sign-off + operator-adoption lines + matching pin + recorded eval-gate PASS). A split commit reddens CI in one direction or the other. This is a future-registration hazard to flag now, not a P1.5 build item.
- Cache TTL choice (5-min vs 1-hour) and the 4-breakpoint budget are unspecified. A per-lead loop that exceeds 5 minutes will lose 5-min-TTL cache hits between leads, and combining anthropic_cache_instructions + tool caching + any explicit CachePoint can exceed the 4-point limit. Needs an explicit, tested decision before claiming cache economics.
- The draft-tier contradiction is unresolved: draft.py:41 pins Opus while the blueprint calls first-pass drafting 'cheaper'. The builder must pick one (move first-pass to Sonnet and reserve Opus for a final/high-risk pass, OR keep Opus and drop the cheap-first-pass claim) and record the decision — shipping the contradiction would make the routing rationale untruthful.
- Consent provenance is only as good as the customers opt-in columns it derives from. If those columns lack a timestamp/source, the first-class Consent record inherits weak provenance and the TCPA/A2P audit trail remains thin even after modeling — may need a schema addition, not just a typed field.
- Pillar B's MIN_SAMPLE / MIN_MARGIN / REPLAN_CAP thresholds are unspecified numerics. Set too low they reintroduce thrash; too high they never fire on real cohorts. These need empirical tuning against seeded cohorts, and the acceptance test's 'majority trust' seed must exceed whatever MIN_SAMPLE is chosen.
- artist_shop_rules / plan.scope enforcement (agui.py:135) is only low-severity-flagged; if left unenforced, a lead can be matched to an artist that is not its own (facts['artist'], agui.py:1268), weakening Pillar D's entity grounding even though nothing sends.
- All 'live run on ladies8391' acceptance criteria require a running Postgres + ANTHROPIC_API_KEY. If the post-delivery verification environment is keyless or DB-less, the caching-is-real and contradiction-fires criteria degrade to unit-level and cannot be adversarially confirmed end-to-end — verification must secure a real environment or explicitly mark those criteria as unverified.
